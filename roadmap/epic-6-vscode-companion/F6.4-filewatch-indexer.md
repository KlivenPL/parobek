# F6.4 — File-watch indexer

- [ ] **Status:** not started
- **Tier:** 1 (Safe) · **Phase:** LATER (VS Code) · **Epic:** 6

## Goal

Keep the embeddings index continuously fresh by watching the workspace for file
changes and incrementally re-embedding only what changed — superseding the
opportunistic NOW refresh in [F4.2](../epic-4-local-rag/F4.2-auto-index.md).

## Mechanism

- Use VS Code's `FileSystemWatcher` (or a persistent daemon the extension owns) to
  observe create/change/delete events, debounce them, and call the **same**
  `lib/index.mjs` incremental update path (per-file re-embed, prune deletions).
- Same on-disk index format and `cwdHash` keying as F4.2, so the CC-side lazy search
  (`local_codesearch`) keeps working unchanged and just finds a warmer index.

## Why LATER (not NOW)

- Continuous watching needs a **persistent process**. Claude Code hooks are one-shot,
  so NOW we only get lazy + detached-on-SessionStart refresh (F4.2). The extension is
  the natural home for a long-lived watcher.

## Feasibility / risk

- Debounce and batch aggressively on large/rapid changes (e.g. branch switches) to
  avoid re-embed storms.
- Coordinate with the F4.2 lazy/opportunistic path so they don't fight over the index
  file (single-writer discipline / lock).

## Depends on

[F4.2](../epic-4-local-rag/F4.2-auto-index.md) (shared indexer core).
