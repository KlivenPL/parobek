# F6.1 — Settings UI

- [ ] **Status:** not started
- **Tier:** 0 (infra) · **Phase:** LATER (VS Code) · **Epic:** 6

## Goal

Replace hand-editing `config.json` with a real settings surface: pick the endpoint
preset, model, embedding model, and **savings tier**, with inline validation.

## Mechanism

- A VS Code settings/webview panel that reads and writes the **same**
  `~/.claude/parobek/config.json` via the existing `lib/config.mjs`
  (`readConfig`/`writeConfig`/`updateConfig`) — no separate config store.
- Tier picker driven by `TIER_PRESETS` (F1.1), showing each tier's name, level, and
  the features it activates (from the feature registry).
- Reuse `configSummaryLines` for a read-only "effective config" view.

## Feasibility / risk

- Must not diverge from the CLI's config semantics — always go through `lib/config.mjs`
  so CLI and UI stay consistent (built-in preset stripping, validation, etc.).

## Depends on

[F1.1](../epic-1-tiered-savings/F1.1-tier-config-foundation.md). The whole `lib/config`
surface is already designed to be a settings backend (see its header comment).
