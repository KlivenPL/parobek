# F4.4 — `local_codesearch` tool

- [ ] **Status:** not started
- **Tier:** 1 (Safe) · **Phase:** NOW · **Epic:** 4

## Goal

An MCP tool that returns the few most relevant code snippets for a query, so Claude
reads far less of a large repo. Pure retrieval — the brain still reasons over the
returned snippets.

## Mechanism

- New tool in the MCP server (F3.1): `local_codesearch(query, k?)`.
- Handler: ensure the index is fresh (lazy build/refresh via F4.2), `embed()` the
  query, compute cosine similarity against stored chunk vectors (plain JS), return the
  top-K as `content` blocks: `path`, line range, and the snippet text.
- Returns *locations + snippets*, not an answer — Claude decides what to open next.
- Defaults: `k = 5`; cap snippet size so the result stays small (the whole point is
  fewer tokens than reading files).

## Files

- **Add:** the `local_codesearch` handler (in `mcp/server.mjs` or `mcp/tools/`),
  using the search-core in `lib/index.mjs` (F4.2).
- **Reuse:** `lib/index.mjs` (search), `lib/provider.mjs` (`embed`).

## Feasibility / risk

- Tier 1: additive retrieval; Claude can always ignore the result and read raw.
- Cold-start latency on the first call (lazy build) — acceptable, and the
  opportunistic `SessionStart` refresh (F4.2) usually warms it first.

## Acceptance & verification

- Integration against a temp repo + `mock-server.mjs`: query for a known snippet,
  assert it ranks in the top-K with the correct path/range.
